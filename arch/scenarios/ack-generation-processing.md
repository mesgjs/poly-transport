# ACK Generation and Processing Scenario

## Overview

This scenario documents how acknowledgments (ACKs) are generated by the receiving side and processed by the sending side to restore flow control budget. ACKs are **transport-level messages** (type 0 headers) that do NOT count toward channel budget. **Critical**: ACKs are fire-and-forget (no ACK-on-ACK), so they use **ring buffer space only**, not transport budget.

## Preconditions

- Transport is started and operational
- Channel is open and data has been exchanged
- Receiver has consumed one or more chunks (marked as ready to ACK)
- Sender has in-flight chunks waiting for acknowledgment

## Actors

- **ChannelFlowControl**: Manages send/receive flow control, generates ACK info, processes ACKs
- **Channel**: Triggers ACK generation based on channel low-water mark
- **Protocol**: Encodes ACK messages into binary format
- **Transport**: Sends ACK messages (transport-level, not channel-level)
- **OutputRingBuffer**: Reserves ring buffer space for ACKs (with `forAck` parameter)

## Key Concepts

### ACK Timing

ACKs are generated **periodically**, not for every chunk:
- **Channel low-water mark**: When channel buffer usage drops below `lowBufferBytes`, generate ACK
- **Batch acknowledgments**: Multiple consumed chunks ACK'd together using range encoding
- **Critical for budget restoration**: ACKs restore sender's channel and transport budgets
- **Ring buffer only**: ACKs use ring buffer space (freed after send), not transport budget

### ACK Message Format

ACK messages are **transport-level** (type 0 headers):
- **Type byte**: 0 (ACK message)
- **Remaining size**: Encoded as `(additionalBytes - 2) >> 1`
- **Flags**: 2 bytes (none defined currently)
- **Channel number**: 4 bytes (local channel number, not remote)
- **Base sequence**: 4 bytes (starting sequence number)
- **Range count**: 1 byte (0-255, should be 0 or odd)
- **Ranges**: Alternating include/skip bytes (each 0-255)
- **Total size**: 13 bytes + range bytes, padded to even (total size is 14 if range bytes is 0 or 1, 16 if range bytes is 3, etc)

### Range-Based Encoding

ACKs use range-based encoding to efficiently acknowledge multiple chunks:
- **Base sequence**: First sequence being ACK'd
- **Include count**: Number of consecutive sequences to ACK (not counting the base)
- **Skip count**: Number of sequences to skip (not ACK'd yet)
- **Pattern**: `[include1, skip1, include2, skip2, ...]`
- **Range count**: Should be 0 (base only) or odd (end with include)
- **Splitting**: Counts >255 split into multiple entries (e.g., 300 → 255, 0, 45)

### Budget Restoration

ACKs restore budget at **two levels**:
1. **Channel budget**: Sender's channel flow control restores in-flight budget
2. **Transport budget**: Sender's transport flow control restores transport budget

Both levels must be restored for sender to continue sending.

## Step-by-Step Sequence

### Part 1: ACK Generation (Receiver Side)

#### Step 1: Chunk Consumption

**Actor**: Application (user code)

**Action**: Application processes received chunk and marks it as consumed.

**Details**:
- User calls `chunk.done()` or `chunk.process(callback)` completes
- Channel calls [`ChannelFlowControl.recordConsumed(seq)`](../../src/channel-flow-control.esm.js:329)
- Chunk marked as consumed (ready to ACK)
- Chunk remains in tracking map until ACK sent

**State Changes**:
- `receivedChunks.get(seq).consumed = true`
- Buffer usage unchanged (still tracked until ACK'd)

#### Step 2: Check Channel Low-Water Mark

**Actor**: Channel

**Action**: After each chunk consumption, check if channel buffer usage dropped below channel low-water mark.

**Details**:
- Calculate channel buffer usage: `flowControl.bufferUsed`
- Compare to channel low-water mark: `bufferUsed < channel.lowBufferBytes`
- If below threshold, trigger ACK generation
- If above threshold, defer ACK generation (batch more chunks)

**Decision Point**:
- **Below channel low-water mark**: Proceed to Step 3 (generate ACK)
- **Above channel low-water mark**: Wait for more chunks to be consumed

**Note**: ACKs use ring buffer space only (not transport budget), so ACK generation is based on channel low-water mark only.

#### Step 3: Generate ACK Information

**Actor**: ChannelFlowControl

**Action**: Generate ACK information for all consumed chunks.

**Method**: [`ChannelFlowControl.getAckInfo()`](../../src/channel-flow-control.esm.js:386)

**Details**:
1. Filter for consumed chunks only (no sorting needed - already in order)
2. If no consumed chunks, return `null` (nothing to ACK)
3. Select base sequence (first consumed chunk)
4. Build range-based encoding:
   - Track consecutive sequences (include count)
   - Track gaps (skip count)
   - Split counts >255 into multiple entries
   - Enforce max 255 total ranges (include + skip)
5. Return `{ baseSeq, ranges }`

**Example**:
- Consumed sequences: `[1, 2, 3, 5, 6, 8]`
- Base sequence: `1`
- Ranges: `[2, 1, 2, 1, 1]` (include 2 after base, skip 1, include 2, skip 1, include 1)
- Interpretation: ACK #1, #2, #3 (base + 2 more), skip #4, ACK #5, #6, skip #7, ACK #8

**State Changes**:
- No state changes (read-only operation)
- Chunks remain in tracking map (not cleared yet)

#### Step 4: Encode ACK Message

**Actor**: Protocol

**Action**: Encode ACK information into binary format.

**Method**: [`Protocol.encodeAckHeaderInto(target, offset, fields)`](../../src/protocol.esm.js:118)

**Details**:
1. Calculate ACK size: `ackHeaderSize(ranges.length)` (computes `toEven(13 + ranges.length)`)
2. Reserve space in output ring buffer (via transport) with `forAck: true` parameter
   - Bypasses `RESERVE_ACK_BYTES` requirement (ACKs don't need extra reservation)
3. Encode ACK header directly into ring buffer:
   - Header-type byte: 0
   - Remaining size: `totalToEncAddl(total)`
   - Flags: 0
   - Channel number: channel ID
   - Base sequence: from ACK info
   - Range count: `ranges.length`
   - Ranges: alternating include/skip bytes
4. Commit reservation to ring buffer

**Data Structures**:
- `target`: VirtualRWBuffer (ring buffer reservation)
- `fields`: `{ flags, channelId, baseSeq, ranges }`

**State Changes**:
- Ring buffer reservation committed (ready to send)

#### Step 5: Send ACK Message

**Actor**: Transport

**Action**: Send ACK message to remote transport.

**Details**:
1. Get buffers from ring buffer: `ring.getBuffers(ackBytes)`
2. Write buffers to underlying transport (e.g., socket, pipe)
3. Consume from ring buffer: `ring.consume(ackBytes)`
4. Zero consumed space (security)

**Budget Impact**:
- Ring buffer space consumed temporarily (freed after send)
- Transport budget unchanged (ACKs are fire-and-forget, no ACK-on-ACK)
- Channel budget unchanged (ACKs are transport-level)

**State Changes**:
- Ring buffer space freed (after send)

**Note**: ACKs are **fire-and-forget** messages (not acknowledged by remote). They use ring buffer space only, not transport budget.

#### Step 6: Clear ACK'd Chunks

**Actor**: ChannelFlowControl

**Action**: Clear ACK'd chunks from tracking.

**Method**: [`ChannelFlowControl.clearAcked(baseSeq, ranges)`](../../src/channel-flow-control.esm.js:345)

**Details**:
1. Remove ACK'd chunks from `receivedChunks` map
2. Subtract bytes from `receivedBytes` counter
3. Return bytes freed

**State Changes**:
- `receivedChunks` map reduced (ACK'd chunks removed)
- `receivedBytes` reduced (buffer space freed)

**Note**: This happens **after** ACK is sent, not before.

### Part 2: ACK Processing (Sender Side)

#### Step 7: Receive ACK Message

**Actor**: Transport

**Action**: Receive ACK message from remote transport.

**Details**:
1. Read bytes from underlying transport (e.g., socket, pipe)
2. Parse header to determine header type (type 0 = ACK)
3. Decode ACK header: [`Protocol.decodeAckHeaderFrom(buffer, offset)`](../../src/protocol.esm.js:273)
4. Extract ACK information: `{ channelId, baseSeq, ranges }`

**Data Structures**:
- `buffer`: VirtualBuffer (input buffer)
- ACK info: `{ channelId, baseSeq, ranges }`

**State Changes**:
- Input buffer consumed (bytes read)

#### Step 8: Find Channel

**Actor**: Transport

**Action**: Look up channel by channel ID.

**Details**:
- Look up channel ID in transport's channel map
- If channel not found, emit `protocolViolation` event (even closed channels are tracked for transport lifetime)

**Decision Point**:
- **Channel found**: Proceed to Step 9
- **Channel not found**: Emit `protocolViolation` event with `UnknownChannel` reason, discard ACK, stop

#### Step 9: Process ACK (Channel Level)

**Actor**: ChannelFlowControl

**Action**: Process ACK and restore channel sending budget.

**Method**: [`ChannelFlowControl.processAck(baseSeq, ranges)`](../../src/channel-flow-control.esm.js:156)

**Details**:
1. Validate ACK sequences:
   - No duplicate ACKs (ACKing same sequence twice is `ProtocolViolationError`)
   - No ACKs beyond assigned (ACKing sequence >= `nextSendSeq` is `ProtocolViolationError`)
2. Remove ACK'd chunks from `inFlightChunks` map
3. Subtract bytes from `inFlightBytes` counter
4. Calculate bytes freed
5. Wake up waiting senders (FIFO order)

**Validation**:
- **Duplicate ACK**: Throws `ProtocolViolationError('DuplicateAck', { sequence, reason })`
- **Premature ACK**: Throws `ProtocolViolationError('PrematureAck', { sequence, nextSendSeq, reason })`

**State Changes**:
- `inFlightChunks` map reduced (ACK'd chunks removed)
- `inFlightBytes` reduced (budget restored)
- Waiting senders awakened (if sufficient budget now available)

**Error Handling**:
- If `ProtocolViolationError` thrown, transport emits `protocolViolation` event
- Transport or channel may close depending on severity and handler decision

#### Step 10: Process ACK (Transport Level)

**Actor**: Transport

**Action**: Restore transport sending budget.

**Details**:
1. Calculate bytes freed (from channel's `processAck()` return value)
2. Subtract from transport in-flight bytes
3. Wake up waiting channels (FIFO order)

**State Changes**:
- Transport in-flight bytes reduced (budget restored)
- Waiting channels awakened (if sufficient budget now available)

**Note**: Transport budget restoration is simpler (no sequence tracking, no validation).

## Postconditions

### After ACK Generation (Receiver)

- Consumed chunks cleared from tracking
- Channel buffer usage reduced
- ACK message sent to remote
- Ring buffer space freed (after send)

### After ACK Processing (Sender)

- In-flight chunks removed from tracking (channel level)
- Channel sending budget restored
- Transport sending budget restored
- Waiting senders/channels awakened (if budget available)

## Error Conditions

### ACK Generation Errors

1. **No consumed chunks**: `getAckInfo()` returns `null`, no ACK generated
2. **Ring buffer full**: ACK generation blocks until space available
   - Ring buffer space freed immediately after send (fire-and-forget)
   - No transport budget required (ACKs are not ACK'd)

### ACK Processing Errors

1. **Duplicate ACK**: `ProtocolViolationError('DuplicateAck')`
   - Transport emits `protocolViolation` event
   - Transport or channel may close depending on handler decision
2. **Premature ACK**: `ProtocolViolationError('PrematureAck')`
   - ACKing sequence that hasn't been sent yet
   - Transport emits `protocolViolation` event
   - Transport or channel may close depending on handler decision
3. **Unknown channel**: ACK for channel that doesn't exist
   - Transport emits `protocolViolation` event with `UnknownChannel` reason
   - Even closed channels are tracked for transport lifetime
   - This indicates malicious or buggy remote transport

## Implementation Notes

### ACK Batching Strategy

**Goal**: Minimize ACK overhead while maintaining responsive flow control.

**Strategy**:
- Generate ACK when channel buffer usage drops below `lowBufferBytes`
- Batch all consumed chunks into single ACK (range-based encoding)
- Don't wait for all chunks to be consumed (may never happen)

**Example**:
- Channel buffer: 64KB
- Low-water mark: 16KB
- Receive 4 chunks of 15KB each (60KB total)
- Consume chunks 1-3 (45KB consumed, 15KB remaining)
- Buffer usage: 15KB (below 16KB threshold)
- Generate ACK for chunks 1-3 (not chunk 4 yet)

### Range Encoding Efficiency

**Goal**: Minimize ACK message size while acknowledging all consumed chunks.

**Strategy**:
- Use base sequence + ranges (not list of all sequences)
- Split large counts (>255) into multiple entries
- End with include count (not skip count)

**Example**:
- Consumed sequences: `[1, 2, 3, 5, 6, 8]`
- Naive encoding: six 4-byte sequence numbers (24 bytes)
- Range encoding: 4-byte base + 5 range bytes (9 bytes)
- Savings: 15 bytes (62%+ reduction)

### Ring Buffer Space for ACKs

**Goal**: Ensure ACKs can always be sent (critical for budget restoration).

**Strategy**:
- ACKs use ring buffer space only (not transport budget)
- ACKs are fire-and-forget (no ACK-on-ACK)
- Ring buffer space freed immediately after send
- Ring buffer reservation with `forAck: true` bypasses `RESERVE_ACK_BYTES` requirement

**Example**:
- Ring buffer: 256KB
- Data ready: 250KB (reserves 250KB + 514 bytes = 250.5KB with `forAck: false`)
- ACK ready: 14 bytes (reserves 14 bytes with `forAck: true`, no extra reservation)
- ACK sent first (priority), space freed immediately
- Data sent next (uses freed space)

### FIFO Waiter Processing

**Goal**: Ensure fairness and prevent starvation.

**Strategy**:
- Process waiters in FIFO order
- Wake up waiters (successively, as they complete) that now have sufficient budget
- Stop when insufficient budget for next waiter

**Example**:
- Waiter + queue: [A: 10KB, B: 5KB, C: 20KB]
- Budget freed: 15KB
- Wake A (10KB), budget remaining: 5KB
- When A completes, wake B (5KB), budget remaining: 0KB
- When B completes, stop (insufficient for C)

### Protocol Violation Handling

**Goal**: Detect and handle malicious or buggy remote transports.

**Strategy**:
- Validate all ACKs (no duplicates, no premature ACKs, no unknown channels)
- Throw `ProtocolViolationError` on violation
- Transport emits `protocolViolation` event for monitoring
- Close channel or transport depending on severity and handler decision

**Example**:
- Receive ACK for sequence 5
- Sequence 5 already ACK'd (duplicate)
- Throw `ProtocolViolationError('DuplicateAck', { sequence: 5 })`
- Transport emits `protocolViolation` event
- Handler decides: close channel, close transport, or ignore (if benign)
